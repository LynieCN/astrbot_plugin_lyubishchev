# AstrBot 柳比歇夫时间管理插件

## 项目简介

这个插件基于柳比歇夫时间管理法，帮助用户记录、整理、回顾自己的时间使用情况，并通过总结与追查功能，尽量把“时间花到了哪里”这件事变得清晰、可追溯。它适合想长期记录日常投入、复盘时间分配、减少“忙了一天但不知道做了什么”这种感觉的人使用。

## 什么是柳比歇夫时间管理法

柳比歇夫时间管理法的核心，不是单纯追求效率，也不是把一天安排得密不透风，而是尽可能真实地记录自己的时间去向。它更强调两件事：1. 记录，即把时间花在哪里尽量记下来，而不是只凭感觉判断自己今天“忙不忙”；2. 复盘，通过周期性的整理和回顾，慢慢看清自己的时间结构，比如哪些事情真正重要、哪些事情消耗很多但产出有限、哪些计划长期没有落实。

## 使用前说明

- 这个插件的设计重点一直是“个人时间记录与复盘”，更偏向个人自用场景。
- 它的目标不是做一个功能特别庞杂的任务管理系统，而是尽量降低记录成本，让时间记录这件事能长期坚持下去。
- 如果你希望的是非常复杂的团队协作、项目流程管理或严格的 GTD 工作流，这个插件可能并不是最合适的选择。

## 可能的不足

- 这个项目最初是从个人需求出发做出来的，所以某些设计会明显偏向我的使用习惯。

- 如果你在使用中遇到问题，欢迎提 issue 或反馈，我会在能力范围内继续完善。

## 隐私与数据提醒

这个插件会记录你的时间内容、标签、分类、项目等信息。  
如果你准备长期使用，建议你自行做好数据备份，并在使用前确认自己对数据存储方式是清楚的。  
如果你会把记录中写入比较私人的内容，也建议注意自己的部署环境和数据安全。

## 详细使用说明

添加时间记录：

```text
/ta <内容>
```

功能命令组：

`/t` 负责查看、修改、总结、查询和计时。  
`/tr` 负责定时总结规则。

### 指令总览

```text
/ta <内容>

/t hp
/t ls [数量|日期|周期|记录ID前缀]
/t ed <记录ID前缀> <新内容>
/t dl <记录ID前缀>
/t ud
/t on <事项>
/t of
/t sm <周期>
/t qy <问题或关键词>
/t st

/tr ls
/tr ad <名称> <自然语言时间>
/tr ad <名称 | cron | 周期 | timezone>
/tr dl <规则ID前缀>
/tr rn <规则ID前缀>
```

### 新增记录

只用 `/ta`。

```text
/ta 09:00-10:30 阅读论文 #科研
/ta 45分钟 处理报销 #行政
/ta 昨天 1.5小时 写周报 category:总结
/ta 2026-03-24 20:00-21:30 背单词 #英语 category:学习
```

批量录入也走 `/ta`：

```text
/ta
09:00-10:30 阅读论文 #科研
10:40-11:10 回邮件 #行政
14:00-15:30 开组会 #工作
```

也支持分号分隔：

```text
/ta 09:00-09:30 站会 #工作；09:30-11:00 写代码 #开发；11:00-11:20 回消息 #行政
```

### 查看与修改

看今天：

```text
/t ls
```

看最近 10 条：

```text
/t ls 10
```

看某一天或某个周期：

```text
/t ls 2026-03-24
/t ls 昨天
/t ls 本周
/t ls 最近7天
/t ls 3.1-3.7
/t ls range:2026-03-01,2026-03-07
```

看单条详情：

```text
/t ls 8f3a
```

修改记录：

```text
/t ed 8f3a 09:00-10:40 阅读论文 #科研 category:学习
/t ed 8f3a 昨天 1小时 处理报销 #行政
```

删除记录：

```text
/t dl 8f3a
```

撤销最近一条有效记录：

```text
/t ud
```

### 计时

开始计时：

```text
/t on 阅读论文
/t on 写代码 #开发 project:NiFe
```

结束计时并自动记账：

```text
/t of
```

典型流程：

```text
/t on 整理实验数据 #科研
/t of
```

### 总结与追查

周期总结：

```text
/t sm day
/t sm 今天
/t sm 本周
/t sm 上周
/t sm 本月
/t sm 上月
/t sm 最近7天
/t sm 14天
/t sm 3.1-3.7
/t sm range:2026-03-01,2026-03-07
```

说明：

- 总结默认只统计实际记录的总时长。
- 如果区间里有 `kind:plan`，会单独提示计划条数和计划时长。

长期记忆追查：

```text
/t qy 我最近时间主要花在哪些事上？
/t qy 帮我找一下上个月和报销有关的记录
/t qy 这周我在 NiFe 项目上花了多少时间？
/t qy 最近有没有连续几天都在写代码？
```

### 定时规则

查看已有规则：

```text
/tr ls
```

自然语言添加规则：

```text
/tr ad 日报 每天22点
/tr ad 日报 每天 22点
/tr ad 周报 每周日21点
/tr ad 月报 每月1号9点
```

高级写法：

```text
/tr ad 每日晚报 | 30 22 * * * | day | Asia/Shanghai
/tr ad 每周周报 | 0 21 * * 0 | week | Asia/Shanghai
/tr ad 双周回顾 | 0 22 * * 0 | custom:14 | Asia/Shanghai
```

手动运行和删除：

```text
/tr rn a1b2
/tr dl a1b2
```

时间段：

时长：


### 常用配置说明

这个插件的配置整体不算复杂。  
如果你只是个人自用，通常只需要先关心下面这几项：

- `default_timezone`
- `analysis_provider_id`
- `embedding_provider_id`
- `record_feedback_enabled`

`default_timezone`
- 默认时区，用于解析相对日期和定时规则。
- 例子：`Asia/Shanghai`

`analysis_provider_id`
用于总结和追查问答的主要模型。
这个配置主要影响：
- /t sm
- /t qy
如果留空，插件会回退到当前会话正在使用的聊天模型。

`embedding_provider_id`
用于长期记忆写入和向量检索的 embedding 模型。
这个配置主要影响：
- 历史记录写入长期记忆
- /t qy 的语义召回能力
如果不配置，插件仍然可以工作，但长期追查会更多依赖文本匹配，效果通常会弱一些。
如果你想认真使用“长期记忆追查”这一块，建议配置。

`rerank_provider_id`
用于 /t qy 的检索结果重排。
这是可选项。
有的话，检索出来的候选结果会再做一次排序，通常能让问答结果更稳一些。
没有也能正常使用，只是最终结果可能没有那么准。
如果你已经配了 embedding_provider_id，并且比较在意 /t qy 的质量，可以再考虑加这个。

`record_scope`
记录隔离粒度。
可选值：
sender：按发送者隔离记录
session：按会话隔离记录
这个插件原本就更偏个人使用，所以一般建议保持 sender

`record_feedback_enabled`

是否在每次成功录入后，额外生成一段反馈。
开启后，新增记录时插件会结合：
- 刚录入的内容
- 最近的聊天
- 最近的时间记录
生成一小段反馈。

优点：
更有陪伴感
记录之后会收到一点回应
有时会顺手提醒你当前节奏
缺点：
会额外消耗模型调用
录入响应会稍微慢一点

`record_feedback_provider_id`

录入反馈使用的模型来源。

这是可选项。
如果不填，插件会优先尝试当前会话使用的模型；这里更像是一个兜底配置。

适合设置的场景：
你想把“录入后的简短反馈”单独交给一个更便宜或更轻量的模型
你不希望录入反馈跟主会话模型完全绑定

注意：
这个配置主要是给“录入反馈”用的，不是给 /t sm 和 /t qy 用的。

`record_feedback_max_recent_records`
生成录入反馈时，最多参考多少条最近记录，默认值通常不需要太大。
这个值越高，反馈时参考的上下文越多，但也可能增加 token 消耗、让反馈变得更啰嗦或引入一些不那么相关的旧信息，一般保持默认或者小幅调整就够了。

`record_feedback_max_recent_chats`
生成录入反馈时，最多参考多少条最近聊天。
作用和上面类似，这个值越高，插件越容易把最近聊天语境带进反馈里，但同时也更容易带来噪音。

`record_feedback_prompt_appendix`
附加到录入反馈里的额外提示词，适合你有明确个人偏好时再使用，比如：

- 希望反馈更简短
- 希望语气更平静
- 希望少一点鼓励，多一点客观提醒
- 希望尽量避免过度解读情绪

示例：`尽量简短，少说套话，不要夸张鼓励。`
如果你没有特别需求，留空即可。

`summary_with_advice`
是否让总结里附带建议。开启后，/t sm 的结果除了做记录归纳，还会更倾向于给出一点复盘和建议。关闭后，总结会更偏“客观汇总”。

`summary_prompt_appendix`
附加到总结 prompt 里的额外要求。适合你已经明确知道自己想要什么风格时再加，比如：

- 更偏简洁
- 更偏复盘
- 更偏指出问题
- 更少空话
示例：`总结尽量直接，优先指出时间分配问题，不要写得太鸡汤。`

`query_answer_with_llm`

/t qy 是否使用 LLM 来组织最终答案。
开启后：
插件会根据检索到的证据，整理成更自然的回答
关闭后：
插件会更直接地返回检索证据本身,更省调用,也更“原始”
如果你更在意可读性，建议开着。
如果你更想看原始命中结果，或者想节省调用，可以关掉。

`max_query_candidates`

长期记忆检索时，最多保留多少条候选证据。
这个值越高：
- 可能召回更多相关内容
- 但也可能把噪音一起带进来
- 同时会增加后续处理成本
一般不建议设得太大。
默认值通常已经够用。

`vector_similarity_threshold`

向量召回阈值。
这个值越高，检索越严格；越低，召回越宽松。
大致可以这样理解：
- 偏低：更容易召回，但可能更杂
- 偏高：结果更严格，但可能漏掉一些本来有用的记录
如果你发现 /t qy 经常召回很多不相关内容，可以适当调高。
如果你发现它老是“查不到”，可以适当调低。

一般建议在一个比较保守的小范围内微调，不要一次改太大。
